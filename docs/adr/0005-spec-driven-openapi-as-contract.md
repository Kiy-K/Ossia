# ADR-0005: OpenAPI as the contract of record (spec-driven development)

**Status:** accepted.
**Date:** 2026-06-22.
**Supersedes:** none.

## Context

A Pydantic model change in `api.py` is silent: nothing checks that the OpenAPI document any client would see today matches the one the clients were written against. Reviewers have to manually diff `app.openapi()` to spot a broken contract, and the diff is usually in a comment, not in CI.

The repo has no `specs/` directory, no `openapi.yaml`, and no contract test. Spec-driven development here means: the OpenAPI document is the contract; it is committed; a test fails when it drifts.

## Decision

1. `specs/SPEC.md` is the narrative spec — vision, scope, NFRs, workflow.
2. `specs/openapi.checked.json` is the machine-readable contract, generated from the live FastAPI app.
3. `tests/test_openapi_drift.py` regenerates the spec and diffs it against the pinned copy. Any drift fails the test with a unified diff and a one-line fix command.
4. `scripts/update_openapi_spec.py` regenerates the pinned spec; reviewers run it when a contract change is intentional.
5. `specs/changelog.md` is the human log of breaking and notable non-breaking changes.

We use raw OpenAPI 3 generated by FastAPI, not a separate spec language (Spectral, Stoplight, Fern). FastAPI is already the source of the routes and the Pydantic models; using anything else would mean keeping two descriptions in sync.

## Consequences

- **Pro:** the contract is executable. `git diff specs/openapi.checked.json` is the contract delta for a PR.
- **Pro:** changing a route without regenerating the spec is a hard test failure, not a code review nit.
- **Pro:** the `Update Spec` script is one command, not a six-step ritual.
- **Con:** we cannot use OpenAPI features that Pydantic + FastAPI don't generate automatically (e.g. webhooks, callbacks). Acceptable for v1.
- **Con:** the pinned JSON is large (~900 lines) and noisy in diffs. Mitigated by sorting keys and indenting uniformly, so diffs are limited to real changes.

## Alternatives considered

1. **Stoplight / Spectral with a hand-written `openapi.yaml`.** Strictly more powerful, but the spec becomes a parallel source of truth that can drift from the code in the opposite direction. Rejected for v1.
2. **Pydantic models as the only contract; no JSON dump.** Would work for Python clients; would lock out JS/Go/Rust clients and any tool that ingests OpenAPI (Postman, Insomnia, contract fuzzers).
3. **TypeSpec / Thrift / gRPC.** Different problem domain; we want a public HTTP contract, not an internal RPC schema.
